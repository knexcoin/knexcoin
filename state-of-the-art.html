Technology Deep-Dive · v1.0

State of the Art

KNEX is a post-quantum Bitcoin fork — re-genesised from block zero with ML-DSA-65 (FIPS 204) signatures enforced at consensus. UMX is a DAG-based payment layer that wraps locked KNEX 1:1000 and settles in under a second. Together they form a two-speed monetary system: KNEX is gravity (100M fixed supply, 5-min blocks, immovable), UMX is velocity (elastic supply, sub-second finality, merchant-friendly). Value flows from circulation (UMX) back toward reserves (KNEX) through DEX conversion and merchant sweeps — what we call economic gravity.

01 · Settlement Layer

Settlement Layer (KNEX) — Bitcoin Fork Regenesis

1.1 Base Inheritance

KNEX is forked from Bitcoin Core but re-genesised — no shared UTXO history with Bitcoin. The Bitcoin Core lineage gives us battle-tested consensus code, P2P networking, block storage, mempool management, and wallet infrastructure. We replaced the cryptography layer and regenesised.

1.2 Post-Quantum Cryptography — ML-DSA-65 (FIPS 204)

Every transaction signature on KNEX is produced by ML-DSA-65 (Module-Lattice Digital Signature Algorithm), the NIST-standardised version of CRYSTALS-Dilithium at security level 3.

PropertyECDSA (Bitcoin)ML-DSA-65 (KNEX)
Security basisElliptic-curve discrete logModule-lattice (LWE + SIS)
Quantum resistanceNo — broken by ShorYes — no known quantum attack
Public key size33 bytes1,952 bytes
Signature size70 bytes (DER)3,309 bytes
Signing speed~100 μs~200 μs
Verification speed~500 μs~70 μs
NIST statusN/AFIPS 204 (2024)

Address derivation pipeline:

seed (256-bit) → SHAKE-256 XOF → ML-DSA-65 keypair
                                      ↓
                               sha256(pk)  →  32-byte pubkey hash
                                                     ↓
                            bech32m(hrp="nex", witness=2, data=hash)
                                                     ↓
                                 nex1z...  (62-char address)

Addresses start with nex1z because bech32m encodes witness version 2 as the character z in the fifth position.

1.3 Output Script Type — WITNESS_V2_PQ

KNEX introduces a new standard output type. Every spendable UTXO on the chain uses this script format:

scriptPubKey:  OP_2  <32-byte pubkey hash>
hex:           52 20 <32 bytes>
  • OP_2 (0x52) signals witness version 2
  • 32-byte push is sha256(ml_dsa_65_public_key) — single-hash, no RIPEMD-160
  • Spending witness: [ sig (3,309-byte ML-DSA sig + sighash byte) | pubkey (1,952 bytes) ]

Verification flow at consensus:

  1. Compute sha256(witness[1]) — the pubkey
  2. Compare against the 32 bytes in the output's scriptPubKey
  3. If match: pass sighash + sig + pubkey to pqclean_ml_dsa_65_ref_crypto_sign_verify()
  4. Accept or reject block

1.4 Consensus Rule — PQ-Only Outputs

A hard consensus rule in src/consensus/tx_check.cpp rejects any transaction that creates a non-PQ output:

for (const auto& txout : tx.vout) {
    TxoutType t = Solver(txout.scriptPubKey, vSolutions);
    if (t != TxoutType::WITNESS_V2_PQ && t != TxoutType::NULL_DATA) {
        return state.Invalid(..., "bad-txout-not-pq");
    }
}

Allowed output types: only WITNESS_V2_PQ and NULL_DATA (OP_RETURN for witness commitments and data anchors). Every legacy Bitcoin script is rejected at the consensus layer:

  • Rejected: P2PK, P2PKH (ECDSA)
  • Rejected: P2SH, bare multisig
  • Rejected: P2WPKH, P2WSH (segwit v0)
  • Rejected: P2TR (taproot / Schnorr)
KNEX is not a hybrid chain. It is 100% post-quantum, enforced by every node.

1.6 Premine Delivery — Block 1 Coinbase Bonus

Bitcoin Core's design makes the genesis-block coinbase unspendable. To deliver the 20M premine into the spendable UTXO set, KNEX adds a one-time bonus to block 1's subsidy via a consensus-level special case in GetBlockSubsidy():

if (nHeight == 1) return era1_subsidy + PREMINE;  // 100 + 20,000,000 NEX

Block 1's coinbase pays 20,000,100 NEX to whichever miner finds it — typically the solo pool's configured payout address. The 20M is then distributed on-chain across 9 founder accounts via normal transactions.

02 · Mining

Mining Infrastructure

2.1 PoW Algorithm — SHA-256d

KNEX uses the same proof-of-work as Bitcoin: double-SHA-256 of the 80-byte block header. This means:

  • Any existing Bitcoin ASIC works immediately (Bitaxe, Antminer, Whatsminer)
  • CPU/GPU miners work too (NMMiner, cpuminer, custom miners)
  • No new hardware needed to participate in consensus

We picked SHA-256d deliberately: Grover's algorithm gives only quadratic quantum speedup, so 256-bit hashes retain 128-bit post-quantum security. The signature scheme is what matters for quantum defense — and that's ML-DSA-65.

2.2 Reference Stratum Pool (NEX Pool)

A 711-line single-file Python asyncio pool ships with the project. It handles the PQ-aware coinbase construction that Bitcoin pools can't produce natively.

03 · Claim System

Claim System — BTC UTXO Snapshot

5M NEX is reserved for a Bitcoin holder claim — any address with a balance in Bitcoin at a specific block height can claim the equivalent NEX by presenting a merkle proof.

3.1 Snapshot Flow

1. bitcoind dumptxoutset at BTC block 940,000
   → produces ~8GB UTXO set binary file
2. extract.py walks the UTXO set
   → groups outputs by address, sums balances
   → builds merkle tree of (address, balance) pairs
3. Merkle root baked into NEX chainparams as FINAL_CLAIM_MERKLE_ROOT
4. NEX node verifies claim transactions via merkle proof

3.2 Claim Transaction Format

Inputs:  [ special "claim marker" input with null prevout ]
Witness: [ merkle proof  | BTC address  | BTC signature of NEX claim address ]
Outputs: [ WITNESS_V2_PQ paying ratio × BTC_balance to claimant's nex1z address ]

The BTC signature proves the claimant owns the original Bitcoin private key. The NEX claim address is signed with the BTC key, then the output on NEX locks to the new PQ key. Each address can claim at most once (enforced via claim index).

04 · Circulation Layer

Circulation Layer (UMX) — DAG Payment Rail

4.1 Architecture — Block-Lattice DAG

UMX doesn't share KNEX's block structure. It runs on Lumero, a block-lattice DAG inspired by Nano but hardened for production commerce:

  • Each account has its own chain
  • Transactions are pairs of blocks (send + receive)
  • Global ordering is NOT required — only per-account ordering
  • Consensus is 2/3 BFT vote-weight on conflicts (Open Representative Voting)

4.2 Transaction Types

TypePurpose
sendDebit sender's chain, reference recipient address
receiveCredit recipient's chain, reference sender's send-block hash
changeUpdate delegated representative (voting)
epochProtocol version marker (soft-fork activation)

4.3 Consensus — Open Representative Voting

  • Nodes delegate voting weight to representatives
  • Representatives vote on conflicts (double-spends)
  • Quorum: 67% of online voting weight confirms within 200-400ms
  • Finality: deterministic, no block-waiting

4.4 Cryptography — Also Post-Quantum

UMX uses the same ML-DSA-65 signatures as KNEX. This means the dual-currency system has one uniform cryptographic surface: a user's 24-word seed derives both their KNEX (witness v2) and UMX (Lumero account) keys deterministically. One backup phrase, two layers.

05 · Bridge

The Bridge — KNEX ↔ UMX

The bridge is what makes this a true dual-currency system rather than two independent chains.

5.1 Conversion Ratio — Fixed

1 NEX ═══════ 1,000 UMX

Not a market rate. A protocol-fixed ratio, enforced by the bridge contract.

5.2 NEX → UMX (Wrap)

1. User locks N NEX to a published bridge address on KNEX chain
2. Bridge observes 6 confirmations (~30 min)
3. Bridge mints 1,000N UMX on Lumero, credits user's account
4. User's KnexPay spend balance increases by 1,000N UMX

5.3 UMX → NEX (Unwrap)

1. User burns 1,000N UMX on Lumero
2. Bridge observes finality (~1 sec)
3. Bridge releases N NEX from reserve to user's KNEX address
4. User's KnexPay reserve balance increases by N NEX

5.4 Reserve Invariant

At any point in time, locked NEX in reserve must equal issued UMX / 1,000.

This is non-negotiable. No fractional reserve. No lending. No leverage. The bridge publishes its reserve accounting on both chains for continuous audit:

  • Lock addresses and balances on KNEX
  • Total UMX supply on Lumero
  • Every mint/burn event with timestamps

5.5 Bridge Controls

  • Multi-sig permissioned signer set (evolves toward decentralisation)
  • Delayed withdrawals above threshold (anti-flash-drain)
  • Rate limits per epoch
  • HSM-backed signing keys
06 · Economic Gravity

Economic Gravity — The Value Flow

KNEX is gravity. UMX is motion. Value moves between them predictably:

Merchant accepts UMX ─► accumulates float ─► sweeps excess to KNEX
                                                       │
                                                       ▼
                                            KNEX reserve grows
                                                       │
                                                       ▼
                                            Settlement layer strengthens
                                                       │
                                                       ▼
                                            UMX circulation supply drops
                                                       │
                                                       ▼
                                            DEX arbitrage rebalances
UMX is a checking account. KNEX is a savings account. KnexPay orchestrates automatic flow between them based on user-defined thresholds.
07 · KnexPay

KnexPay — Consumer Wallet

7.1 Architecture — iOS Native + WKWebView Hybrid

  • Swift shell, WebKit rendering engine
  • Native bridge locked to packaged app origin (no arbitrary URL loads)
  • Dual balance UI: Reserve (NEX) + Spend (UMX)
  • Real-time updates via WebSocket to node

7.2 Key Storage

  • Keys live only in iOS Keychain (biometric-protected, Secure Enclave on A13+)
  • AES-256-GCM encryption at rest
  • Never synced to iCloud
  • Never readable by JavaScript layer

7.3 Auto-Float Logic

Users set three thresholds:

ControlBehavior
Min spend balanceBelow this, auto-wrap from reserve
Target spend balanceAuto-refill aims for this number
Max spend floatAbove this, auto-sweep back to reserve

This makes UMX behave like a tightly-managed checking account, with surplus flowing automatically to KNEX savings.

7.4 NFC Tap-to-Pay

  • iPhone's NFC stack (Core NFC) reads SmartBills (NDEF bearer notes)
  • Each tap verifies a CMAC (Cipher-based MAC) signed with the bill's SUN/SDM key
  • Server-side validation prevents replay attacks
  • Supports both personal transfers and merchant point-of-sale

09 · Security

Security Model — Defense in Depth

9.1 Layer 1 — Cryptography

  • ML-DSA-65 signatures (quantum-proof)
  • SHA-256d PoW (Grover-resistant at 256 bits)
  • BIP-39 seed entropy (256-bit root)

9.2 Layer 2 — Consensus

  • PQ-only output rule rejects ECDSA outputs at consensus
  • Block weight caps prevent DoS
  • BIP34 coinbase height enforcement
  • Segwit witness commitment required in coinbase

9.3 Layer 3 — Network

  • P2P connects over NEX magic bytes (prevents Bitcoin cross-contamination)
  • TLS-optional RPC binding (localhost default)
  • Mempool-size caps prevent flood

9.4 Layer 4 — Application

  • CLI wallet signs 100% locally — private keys never touch the node
  • KnexPay keys in iOS Keychain only
  • Bridge reserve invariant continuously published

9.5 Layer 5 — Operational

  • Multi-sig bridge signer
  • Delayed large withdrawals
  • Public reserve accounting
  • On-chain claim merkle root (immutable once set)
